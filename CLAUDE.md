# CLAUDE.md

旨在减少大语言模型（LLM）常见编码错误的常规行为指南。可根据需要与具体项目的指令进行合并。

**权衡取舍：** 这些指南更倾向于“谨慎”而非“速度”。对于微不足道的简单任务，请酌情衡量。

## 1. 动笔前先思考 (Think Before Coding)

**不要盲目假设。不要隐瞒疑惑。把权衡方案摆在台面上。**

在开始实现之前：

- 明确写出你的假设。如果不确定，请主动提问。
- 如果存在多种理解方式，请把它们都列出来——不要默默地替用户做决定。
- 如果有更简单的方法，请提出来。在合理的情况下，可以对不合理的需求进行驳回。
- 如果有任何地方不清楚，请立即停下。指出让人困惑的点，并提问。

## 2. 简单至上 (Simplicity First)

**用最少的代码解决问题。拒绝任何投机性/推测性的代码。**

- 绝不添加超出需求范围的功能。
- 绝不为只使用一次的代码做抽象封装。
- 绝不提供未被要求的“灵活性”或“可配置性”。
- 绝不为不可能发生的场景编写错误处理逻辑。
- 如果你写了 200 行代码，而实际上 50 行就能搞定，请重写。

扪心自问：“资深工程师会觉得这写得太复杂了吗？”如果是，请精简。

## 3. 精准修改 (Surgical Changes)

**只动必须要动的地方。只清理自己留下的痕迹。**

当修改现有代码时：

- 不要去“顺手改进”周边的代码、注释或格式。
- 不要重构没有损坏的东西。
- 必须匹配既有的代码风格，即使你换个写法能写得更好。
- 如果注意到无关的死代码（无用代码），请提出来——不要擅自删除。

当你的修改导致其他代码失去作用时：

- 仅移除因**你自己的修改**而变得不再被使用的 import、变量或函数。
- 除非被要求，否则不要移除原本就存在的死代码。

**检验标准：** 修改的每一行代码，都必须能直接追溯到用户的需求。

## 4. 目标导向执行 (Goal-Driven Execution)

**明确成功标准。不断循环迭代，直到验证通过。**

将任务转化为可验证的目标：

- “添加校验” → “为非法输入编写测试，然后让测试通过”
- “修复 Bug” → “编写一个能复现该 Bug 的测试，然后让测试通过”
- “重构 X” → “确保重构前后的测试都能通过”

对于多步骤的任务，请陈述一个简要的计划：

```
1. [步骤] → 验证方式: [检查项]]
2. [步骤] → 验证方式: [检查项]]
3. [步骤] → 验证方式: [检查项]]
```

清晰的成功标准能让你独立进行循环验证。模糊的标准（如“让它跑起来”）则需要频繁的沟通澄清

---

**如何判断这些指南起到了作用：** 代码 Diff 中不必要的改动变少了；因为想得太复杂而重写代码的情况变少了；在动手实现之前提出了澄清问题，而不是在犯错之后才去弥补。
